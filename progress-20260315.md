# Progress Log - 2026-03-15

## Inventor Adapter: First Live Testing Against Inventor 2025

### Summary

Ran the Inventor round-trip test suite (`tests/test_inventor_roundtrip.py`, 54 tests) against a live Autodesk Inventor Professional 2025.2 instance for the first time. All 54 tests initially failed. Through iterative diagnosis and fixing, brought the suite to 54/54 passing.

### Environment Setup

- Python 3.10.19 required (project uses `float | None` union syntax); the base Anaconda env has 3.9, so the `sketch_bridge` conda env was used.
- Installed `pywin32` and `pytest` into `sketch_bridge` env.
- Fixed test import: `from adapter_inventor` → `from morphe.adapters.inventor` (no `adapter_inventor` package exists).

### Files Modified

- `morphe/adapters/inventor/adapter.py` — Major fixes to COM API calls
- `morphe/adapters/inventor/vertex_map.py` — Rewrote entity type detection
- `tests/test_inventor_roundtrip.py` — Fixed import path, constructor args, type filtering

### Root Causes and Fixes

**1. Document type constant (adapter.py:142)**
- `kPartDocumentObject` was `12163`; Inventor 2025 uses `12290`.
- Discovered by checking `doc.DocumentType` on an existing open document.

**2. Ellipse creation API (adapter.py:~470)**
- Old: `SketchEllipses.Add(centerPoint, majorAxisPoint, minorAxisPoint)` — fails with "float() argument must be a string or a real number, not 'CDispatch'"
- New: `SketchEllipses.Add(centerPoint, majorAxisUnitVector, majorRadius, minorRadius)`
- The API takes a `UnitVector2d` for axis direction and separate scalar radii.

**3. EllipticalArc creation API (adapter.py:~530)**
- Old: passed start/end points — same CDispatch error.
- New: `SketchEllipticalArcs.Add(center, majorAxisUnitVector, majorRadius, minorRadius, startAngle, sweepAngle)`
- Takes parametric angles, not point coordinates.

**4. Dimension constraint API names (adapter.py:~850-890)**
- `AddRadiusDimension` → `AddRadius`
- `AddDiameterDimension` → `AddDiameter`
- `AddLinearDimension` does not exist; replaced with `AddTwoPointDistance` using the line's start/end sketch points.

**5. `AddTwoPointDistance` requires orientation enum (adapter.py:~810-880)**
- Old code passed 3 args; Inventor 2025 requires 4: `(pt1, pt2, orientationEnum, textPoint)`.
- Enum values discovered via type library: `kHorizontalDim=19201`, `kVerticalDim=19202`, `kAlignedDim=19203`.
- Distance uses aligned (19203), DistanceX uses horizontal (19201), DistanceY uses vertical (19202).

**6. `doc.Update()` needed after dimension changes (adapter.py:~218)**
- Setting `dim.Parameter.Value` doesn't update geometry until `doc.Update()` is called.
- Added `self._document.Update()` at end of `load_sketch()` after all constraints.

**7. Coincident constraint: `Merge` not `AddCoincident` (adapter.py:~655)**
- `AddCoincident(point, entity)` constrains a point to lie on an entity — it does NOT merge two points.
- For point-to-point coincidence, `SketchPoint.Merge(otherSketchPoint)` is the correct API.

**8. Fixed/Ground on PointRef (adapter.py:~810)**
- When `Fixed(PointRef(line_id, PointType.START))` is used, must ground the specific sketch point, not the parent line entity.

**9. Point export filtering — COM identity problem (adapter.py:~290-380)**
- `_is_dependent_point` relied on `SketchPoint.DependentObjects` which doesn't exist in Inventor 2025.
- Python `id()` is unstable for COM wrappers (new wrapper object each access).
- `repr(_oleobj_)` address extraction was tried but also unstable across collection vs property access.
- **Solution**: Collect all owned sketch point COM objects from geometry, then iterate SketchPoints collection comparing via `_oleobj_` equality (which correctly identifies COM identity).
- Uses index-based filtering: builds `set[int]` of 1-based collection indices to skip.

**10. Spline export missing + fit point API (adapter.py:~290, ~1000)**
- Export had no spline section — added iteration over `sketch.SketchSplines`.
- Spline fit points: API is `spline.FitPoint(i)` (method call) and `spline.FitPointCount`, not `spline.FitPoints.Item(i)`.

**11. Solver status detection (adapter.py:~1320)**
- `sketch.IsFullyConstrained` doesn't exist via late-binding COM.
- Replaced with per-entity `ConstraintStatus` property check: `51713` = fully constrained, `51714` = under constrained.

**12. Vertex map entity type detection (vertex_map.py — full rewrite)**
- `type(entity).__name__` returns `'CDispatch'` for all COM late-binding objects.
- Replaced with duck-typing: checks property presence (`StartSketchPoint`, `CenterSketchPoint`, `Radius`, `MajorRadius`, `FitPoints`, etc.) to determine entity kind.
- Also improved `_same_point()` to try `_oleobj_` equality before falling back to coordinate comparison.

**13. Test fixes (test_inventor_roundtrip.py)**
- `DistanceX(pt1, pt2, value=80)` → `DistanceX(pt1, 80, pt2=pt2)` — constructor signature is `(pt, value, pt2=None)`.
- Several tests that iterated `exported.primitives.values()` assuming all were lines/circles now filter by type with `isinstance`.

### Key Decisions

- **Duck-typing over type library**: Chose duck-typing for entity identification rather than early-binding (generating type library with `makepy`). Early-binding would give proper type names but adds complexity and version coupling. Duck-typing works reliably and is version-agnostic. **Alternative rejected**: Using `entity.Type` numeric constants — these vary across Inventor versions and are hard to maintain.

- **`_oleobj_` equality for COM identity**: After trying `id()`, `repr()` address parsing, and `IUnknown` queries, found that `_oleobj_` equality comparison is the only reliable approach. It correctly identifies the same underlying COM object across different Python wrapper instances. However, it's not hashable, so we use a list-based matching approach with index tracking.

- **`Merge` vs `AddCoincident` for point coincidence**: Inventor's `AddCoincident` constrains a point to an entity (point-on-curve), not point-to-point merging. `SketchPoint.Merge()` is the correct API for making two endpoints share the same location — it's the programmatic equivalent of dragging one endpoint onto another in the UI.

### Numeric Results

- **Final test results**: 54/54 passed in 95.27 seconds
- **Progression**: 0/54 → 24/54 → 28/54 → 34/54 → 51/54 → 54/54

### Next Steps

- Consider adding early-binding COM support (`win32com.client.gencache.EnsureDispatch`) for better performance and type safety, though duck-typing works well enough for now.
- The `_collect_owned_sketch_point_indices` method does O(n*m) comparisons; could be optimized if sketches get very large.
- Spline round-trip currently uses fit points only (start/end match but interior points shift slightly due to Inventor's spline fitting). True B-spline control point export would need `SplineDefinition` API.

## Code Review Fix Plan Implementation

### Summary

Implemented the "Fix Plan: Code Defects and Spec Drift" across three commits corresponding to three phases.

### Phase 1: Core library fixes (commit 7228199)

**Files modified**: `morphe/validation.py`, `morphe/primitives.py`, `morphe/serialization.py`, `morphe/constraints.py`, `morphe/document.py`

Changes:
- **validation.py**: Added `_validate_ellipse()` and `_validate_elliptical_arc()` with dispatch in `_validate_primitive()`. Checks: non-finite center, positive radii, major >= minor, non-finite params, start_param != end_param.
- **primitives.py**: Added `ValueError` guard in `Ellipse.eccentricity` and `Ellipse.focal_distance` when `minor_radius > major_radius` (previously crashed with opaque `math domain error`). Added `NotImplementedError` in `Spline.evaluate()` when `is_rational` is True (previously silently ignored weights).
- **serialization.py**: `_parse_point()` now raises `ValueError` on truncated data instead of silently returning origin. `dict_to_primitive()` now requires mandatory geometry fields (start/end for Line, center/start_point/end_point for Arc, etc.) — raises `ValueError` if missing instead of fabricating geometry with defaults.
- **constraints.py**: Changed `_generate_id()` from `uuid4()[:8]` (32 bits) to `uuid4()[:12]` (48 bits). Birthday collision threshold moves from ~65K to ~16M.
- **document.py**: `add_primitive_with_id()` now validates that ID prefix matches primitive type (e.g., rejects assigning "A0" to a Line). `get_point()` now raises `NotImplementedError` for `PointType.ON_CURVE` instead of falling through to opaque `ValueError`.

### Phase 2: Adapter fixes (commit 34f46f2)

**Files modified**: `morphe/adapters/solidworks/adapter.py`, `morphe/adapters/freecad/adapter.py`, `morphe/adapters/fusion/adapter.py`, `morphe/adapters/inventor/adapter.py`

Changes:
- **SolidWorks `_add_concentric`**: Arc branch was reading `geom2['radius']`, `geom2['start_angle']`, `geom2['end_angle']` which don't exist in the stored geometry dict. Fixed to compute radius and angles from stored `center`, `start`, `end` points.
- **FreeCAD `_add_tangent`**: Was using `pt[1]` (the connection point's vertex index) for both geometries. Fixed to use 3-arg Tangent form: identifies which geometry owns the connection point and passes `(geo_with_point, vertex, other_geo)`.
- **FreeCAD/Fusion CCW**: `ccw = end_angle > start_angle` fails for arcs crossing 0/2π. Both FreeCAD's `ArcOfCircle` and Fusion's `SketchArc` parameterize sweeps as always CCW. Changed to `ccw = True`. Same fix for Fusion elliptical arcs.
- **SolidWorks DOF**: All sketch points were counted as 2 DOF each, but most are segment endpoints already included in per-segment DOF (4 for line, 5 for arc). Fixed to subtract segment endpoint count before adding standalone point DOF.
- **FreeCAD `_geo_to_prim_type`**: Silently defaulted unrecognized geometry to `Line`. Changed to raise `ValueError`.
- **Inventor spline**: Import now logs a warning when importing non-fit splines (NURBS fidelity lost). Export now sets `is_fit_spline=True` since we read fit points.
- **SolidWorks elliptical arc CCW**: Was hardcoded `ccw=True`. Now computes actual direction via cross product of (center→start) × (center→mid).
- **SolidWorks constraint dedup**: Key `(rel_type, segment_id)` was too aggressive — dropped valid constraints of the same type on the same segment with different partners. Changed to `(rel_type, tuple(sorted(all_involved_ids)))`.

### Phase 3: SPECIFICATION.md sync (commit a996afa)

Changes:
- Fixed `ELLIPTICAL_ARC` prefix from `"EA"` to `"e"` to match code.
- Added 5 missing `CONSTRAINT_RULES` entries: COLLINEAR, DISTANCE_X, DISTANCE_Y, SYMMETRIC, MIDPOINT.
- Added Ellipse/EllipticalArc to `_next_index` and `add_primitive` prefix map.
- Added Ellipse/EllipticalArc branches to `primitive_to_dict`. Fixed `constraint_to_dict` to show conditional field inclusion (omit value/inferred/confidence when default).
- Added MIDPOINT to EllipticalArc `get_valid_point_types`.
- Added sketch name header, `[C]` construction marker, and Ellipse/EllipticalArc formatting to `_describe_primitive`.
- Added reference validation to `add_constraint` (raises `KeyError` for non-existent elements).

### Key Decisions

- **FreeCAD/Fusion CCW = always True**: Rather than computing cross products (which would be more general), relied on the documented API behavior that both FreeCAD and Fusion parameterize arc sweeps as CCW. This is simpler and avoids depending on midpoint computation. The cross-product approach was used for SolidWorks elliptical arcs where the API convention is less clear.
- **Tangent 3-arg form**: Chose FreeCAD's 3-argument `Tangent(geo, vertex, other_geo)` over the 4-argument form because it correctly establishes tangency at a specific point without requiring vertex enumeration on the other geometry. Falls back to 4-arg form when the connection point doesn't belong to either referenced geometry.
- **Dedup key**: Changed from `(rel_type, segment_id)` to `(rel_type, tuple(sorted(all_involved_ids)))`. This allows multiple constraints of the same type on the same segment with different partners while still deduplicating the same constraint seen from multiple segments.
