# Progress — 2026-04-26

## Session: C++17 reference implementation, Phase 1 (skeleton)

### Summary

Started a C++17 reference implementation of the Morphe schema under
`cpp/`. The goal is a standalone library that adopters can vendor or
`add_subdirectory` into a CMake project, enabling Morphe adoption from
non-Python toolchains (native CAD plugins, kernels, FFI consumers).
The implementation does not link against, embed, or call into the
Python package; the only point of contact is a shared JSON conformance
corpus to be added in Phase 5.

Plan document: `~/.claude/plans/i-m-thinking-about-adding-bubbly-wave.md`.

### Files created

- `cpp/CMakeLists.txt` — CMake 3.16+, C++17, `morphe` INTERFACE target,
  `morphe::morphe` alias. Test target `morphe_tests` gated on
  `MORPHE_BUILD_TESTS` (default ON when top-level, OFF when consumed via
  `add_subdirectory`).
- `cpp/include/morphe/morphe.hpp` — umbrella header with version
  constants. Subsequent phases populate it.
- `cpp/tests/test_main.cpp` — doctest entry point.
- `cpp/tests/test_smoke.cpp` — version-constant check + nlohmann/json
  trivial round-trip, to prove the toolchain and both vendored
  dependencies link cleanly.
- `cpp/third_party/nlohmann/json.hpp` — vendored single header
  (nlohmann/json v3.11.3, MIT).
- `cpp/third_party/doctest/doctest.h` — vendored single header
  (doctest v2.4.11, MIT, tests only).
- `cpp/README.md` — build instructions, consumption pattern, vendored
  dep manifest, phase status table.

### Key decisions

- **Language: C++17.** Chosen over pure C99/C11 and over a C++ core +
  C ABI shim. Rationale: the primary readers of a "reference
  implementation" are people figuring out the format; C++17's
  `std::variant` and `std::optional` map directly to the schema's tagged
  primitives and omit-when-default fields, making the source itself a
  precise specification. A C ABI wrapper can be added later without
  changing the core if FFI demand emerges. **Rejected**: pure C
  (≈3× the code, less ergonomic as documentation) and dual C++/C
  surfaces (two surfaces to maintain on day one).
- **JSON library: vendor `nlohmann/json` single header at v3.11.3.**
  Drop-in, no CMake `FetchContent` step, builds offline on a fresh
  machine. **Rejected**: `FetchContent` (network on first configure;
  adds a CMake dependency adopters may not want), `find_package`
  (pushes setup burden onto adopters and varies by distro/Windows).
- **Test framework: doctest v2.4.11.** Single-header, fast, no separate
  build target needed. Same vendoring pattern.
- **Library is INTERFACE (header-only) at this stage.** Becomes a
  static library if and when implementation `.cpp` files appear (likely
  in Phase 3 for serialization helpers; the rest can stay header-only).
- **`tests/conformance/` will live at the repo root, not under `cpp/`.**
  The corpus is consumed by both Python and C++, so it is canonical
  schema test data, not C++-specific.

## Session: C++17 reference implementation, Phase 2 (core types)

### Summary

Implemented the full Morphe data model in C++17, mirroring
`morphe/types.py`, `morphe/primitives.py`, `morphe/constraints.py`,
and `morphe/document.py`. No serialization or validation yet — those
are Phases 3 and 4. Headers are organized one per concept:

- `types.hpp` — `Point2D`, `Vector2D`, `PointType`, `PointRef`,
  element-prefix constants.
- `primitives.hpp` — `PrimitiveMeta`, the seven primitive structs
  (`Line`, `Arc`, `Circle`, `Point`, `Spline`, `Ellipse`,
  `EllipticalArc`), `Primitive` variant, free-function accessors
  (`meta_of`, `id_of`, `type_tag`, `id_prefix`, `valid_point_types`).
- `constraints.hpp` — `ConstraintType` (19 enumerators, 20 wire
  values — see footgun below), `ConstraintStatus`, `ConstraintRef`
  variant, `RefKind` enum, `ConstraintRules` struct, and
  `SketchConstraint` aggregate with `get_element_ids()`.
- `document.hpp` — `SolverStatus`, `SketchDocument` class with
  `add_primitive` (auto-ID), `add_primitive_with_id`,
  `find_primitive`, `add_constraint`.
- `src/constraints.cpp` — the `CONSTRAINT_RULES` table, ported
  verbatim from Python. Now a static library (was INTERFACE).

### Files created

- `cpp/include/morphe/types.hpp`
- `cpp/include/morphe/primitives.hpp`
- `cpp/include/morphe/constraints.hpp`
- `cpp/include/morphe/document.hpp`
- `cpp/src/constraints.cpp` — only `.cpp` so far; rules table needs
  out-of-line definition to keep it singleton.
- `cpp/tests/test_types.cpp`
- `cpp/tests/test_primitives.cpp`
- `cpp/tests/test_constraints.cpp`
- `cpp/tests/test_document.cpp`

### Files modified

- `cpp/include/morphe/morphe.hpp` — umbrella now includes the four
  concept headers.
- `cpp/CMakeLists.txt` — switched `morphe` from `INTERFACE` to
  `STATIC` (now needs `src/constraints.cpp`); added the four new test
  sources; warnings flags moved onto the library target as well.

### Key decisions

- **`Primitive = std::variant<Line, Arc, Circle, Point, Spline,
  Ellipse, EllipticalArc>`.** Free-function accessors (`meta_of`,
  `type_tag`, `id_prefix`, `valid_point_types`) operate over the
  variant via `std::visit`. This keeps each primitive a plain struct
  with public data members — the source reads as documentation of the
  schema. **Rejected**: a virtual base class. Adds vtables for no
  benefit, and the wire format is closed (the spec lists exactly seven
  primitives), so a sealed variant is the right shape.
- **Insertion-ordered `std::vector<Primitive>` instead of a map.**
  Python's `SketchDocument.primitives` is a dict — insertion-ordered
  since 3.7, and the serializer iterates it via `.values()`. Using a
  `std::map` would re-sort by key; using `std::unordered_map` would
  randomize. A `std::vector` plus `find_primitive(id)` linear lookup
  matches Python's iteration order exactly. Linear lookup is fine at
  the scale Morphe targets (sketches with hundreds of primitives, not
  millions).
- **`PrimitiveMeta` as a nested struct accessed via `.meta`, not
  inheritance.** Primitives stay aggregate-initializable (no
  constructor noise); `PrimitiveMeta` can be reused without slicing
  worries; `meta_of(p)` over the variant is one visitor instead of
  per-type accessors. **Rejected**: inheriting from `PrimitiveMeta`
  (would prevent aggregate initialization in C++17 and complicate the
  variant).
- **Element prefix for `EllipticalArc` is lowercase `'e'`.** Matches
  Python's `ElementPrefix.ELLIPTICAL_ARC = "e"`. Unit-tested as a
  guard against silent regression.
- **`ConstraintType::Midpoint` serializes as `"midpoint_constraint"`,
  not `"midpoint"`.** This is a footgun in the Python source
  (`morphe/constraints.py:40`); the C++ enumerator is named `Midpoint`
  but its wire string is `"midpoint_constraint"`. Pinned by an
  explicit test and called out in code comments at both the enum and
  the converter.

### What was intentionally NOT done

- **No JSON yet.** Phase 3 introduces `to_json` / `from_json` /
  `save` / `load` and the Python-equivalent omit-when-default
  encoding rules.
- **No validation.** Phase 4. The structures support validation
  (e.g., `Arc::radius()` for the consistency check, `Line::length()`
  for the degenerate-line check), but no rule logic is implemented.
- **Skipped computational helpers from Python:** spline `evaluate()`
  (De Boor's algorithm), ellipse `circumference` (Ramanujan
  approximation), `eccentricity`, `focus1/focus2`, arc
  `sweep_angle`, etc. These are kernel-level operations not part of
  the schema and not needed for (de)serialization or validation.
  Adopters embedding Morphe will plug it into their own kernel.
  Documented this scope choice in `cpp/README.md` (deferred until
  Phase 5 when README gets a real adoption section).
- **No constraint convenience constructors** (`Coincident(pt1, pt2)`,
  `Tangent(...)`, etc., which Python provides). C++ aggregate
  initialization with `{...}` covers the same ground; adding
  free-function builders is sugar that can come later if adopters
  ask. The default-constructed `SketchConstraint{}` plus member
  assignment is fine for tests and corpus generation.

## Session: C++17 reference implementation, Phase 3 (serialization)

### Summary

Implemented full JSON (de)serialization mirroring
`morphe/serialization.py`. Public API (in
`include/morphe/serialization.hpp`):

- `std::string to_json(const SketchDocument&, int indent = 2)`
- `SketchDocument from_json(std::string_view)`
- `void save(const SketchDocument&, const path&, int indent = 2)`
- `SketchDocument load(const path&)`

Implementation in `src/serialization.cpp` uses
`nlohmann::ordered_json` (insertion-ordered) for output to match
Python's dict-iteration key order; `nlohmann::json` for input where
key order is irrelevant.

### Files created

- `cpp/include/morphe/serialization.hpp`
- `cpp/src/serialization.cpp` (~290 LOC)
- `cpp/tests/test_serialization.cpp` (22 test cases / 113 assertions)

### Files modified

- `cpp/CMakeLists.txt` — added `src/serialization.cpp` to the
  `morphe` static library and `tests/test_serialization.cpp` to
  the test target.

### Key decisions

- **`nlohmann::ordered_json` for output, plain `nlohmann::json` for
  input.** The default `nlohmann::json` is a thin wrapper around
  `std::map`, which sorts object keys alphabetically — that would
  break parity with Python's dict insertion order. `ordered_json`
  uses `nlohmann::ordered_map` instead. **Rejected**: writing a
  custom serializer that emits keys in a fixed order. The
  `ordered_json` solution is one type alias and zero policy code,
  vs. ~50 LOC of bespoke ordering and a maintenance burden as
  fields evolve.
- **Encoding policy: omit optional fields at default values, do not
  emit `null` placeholders.** Diverges from `SPECIFICATION.md`
  §13.2's example (which shows `value: null, inferred: false,
  confidence: 1.0`) but matches Python's actual encoder
  (`morphe/serialization.py` lines 100-104, 277-294, 281-282 etc.).
  Pinned by 7 explicit "omitted when default" tests.
- **Decoder is permissive on missing optional fields, strict on
  unknown enum strings.** Matches Python: `data.get("source")`,
  `data.get("confidence", 1.0)` — silent defaults; vs.
  `SolverStatus(status_str)`, `ConstraintType(...)` — throw on
  unknown.

### What was intentionally NOT done

- **No conformance corpus yet.** Phase 5. The cross-language parity
  check above was a one-off to confirm the format is right; the
  corpus turns this into a checked-in, regenerable test fixture.
- **No JSON Schema document published.** The C++ source is now an
  independent reading of the schema; if the user wants a formal
  JSON Schema for IDE/editor validation, that is a separate
  artifact.
- **No round-trip byte-equivalence with Python.** Float formatting
  differs subtly between nlohmann/json's shortest-round-trip and
  Python's `repr(float)` (e.g., `1.5707963267948966` may render
  differently). Semantic equivalence is enforced via parsed-value
  comparison; Phase 5's corpus tests use that approach. Byte
  equivalence is not a goal.
- **No support for the §13.2 example's verbose encoding.** The C++
  encoder will not emit `value: null` etc. If a user supplies a
  document encoded in that older style, the C++ decoder accepts
  it (null fields are treated as absent), but C++ output will use
  the omit-when-default form. This matches Python's behavior.

## Session: C++17 reference implementation, Phase 4 (validation)

### Summary

Ported `morphe/validation.py` to C++ verbatim, including the
severity-tagged issue model (Error / Warning / Info) and stable
machine-readable error codes (e.g., `ARC_RADIUS_INCONSISTENT`,
`SPLINE_INVALID_KNOTS`). Public API:

- `morphe::ValidationSeverity` enum
- `morphe::ValidationIssue` struct (severity, element_id, message, code)
- `morphe::ValidationResult` class (issues vector + `is_valid()`,
  `has_warnings()`, `errors()`, `warnings()`)
- `morphe::validate(const SketchDocument&, double tolerance = 0.001)`
- `morphe::default_tolerance` constant (0.001 mm = 1 micron)

### Files created

- `cpp/include/morphe/validation.hpp`
- `cpp/src/validation.cpp` (~250 LOC)
- `cpp/tests/test_validation.cpp` (34 test cases / 50+ assertions)

### Files modified

- `cpp/include/morphe/morphe.hpp` — umbrella now includes
  `validation.hpp`.
- `cpp/CMakeLists.txt` — added `src/validation.cpp` to library and
  the test source.

### Rules ported

**Per-primitive (errors/warnings):**
- All: empty ID, confidence outside [0,1].
- Line: zero length below tolerance, non-finite coordinates.
- Arc: inconsistent radius (|r_start - r_end| > tolerance), zero
  radius, degenerate (start ≈ end → use Circle), non-finite coords.
- Circle: non-positive radius, tiny radius (warning), non-finite.
- Point: non-finite coordinates.
- Spline: degree < 1, insufficient control points (need degree+1),
  non-finite control points, non-monotonic knot vector, knot length
  mismatch (warning), weight count mismatch, non-positive weight,
  non-finite weight.

**Per-constraint:**
- Empty ID (warning); too few/many references; missing required
  value for dimensional constraints; non-finite value; negative
  value for `Length`/`Radius`/`Diameter`/`Distance` (NOT for
  `Angle`); non-existent reference; invalid `PointType` for the
  referenced primitive; `CONTROL` missing index; `CONTROL` index
  out of range; bad `connection_point`; constraint confidence
  outside [0,1] (warning).

**Cross-document:**
- Duplicate constraint IDs (warning).

**Intentionally NOT validated** (matching Python):
- `Ellipse` and `EllipticalArc` have no primitive-level rules. The
  Python validator dispatches only on `Line`/`Arc`/`Circle`/
  `Point`/`Spline`. Mirrored exactly; documented in code comment.
- `ConstraintRules.ref_kinds` is NOT enforced. The Python validator
  checks reference COUNT and the validity of `PointType` per
  primitive, but does not check that the referenced primitive's
  type matches the rule's allowed kinds (e.g., a `Horizontal`
  constraint applied to a `Circle` passes validation in both
  implementations). This is a shared gap, not a porting omission.

### Numeric constants preserved verbatim

- `default_tolerance = 0.001` mm — used for line min length, arc
  radius consistency, arc zero-radius, arc degenerate, circle tiny
  radius, and the spline knot tolerance. Single named constant in
  `validation.hpp` so it can't drift.

### Bug caught mid-session by stricter validation

The "duplicate constraint IDs are a warning" test initially failed:
it constructed a default-initialized `m::Line{}` (start=end=(0,0))
just to satisfy reference resolution, but the new validator
correctly flagged it as `LINE_ZERO_LENGTH`. Fix: gave the line
non-degenerate endpoints `(0,0) → (1,0)`. Lesson worth keeping: a
working validator forces every test fixture to be a *valid* sketch,
not just a *constructed* sketch — this is exactly the kind of
unintended-state-as-test-noise that a strict validator is supposed
to catch, and it caught itself.

### What was intentionally NOT done

- **No constraint-kind enforcement.** As above, a `Horizontal`
  constraint on a `Circle` still passes — gap shared with Python.
- **No semantic constraint checking** (e.g., would the constraint
  system produce a consistent solution? Are angles between parallel
  lines well-defined? Is a spline knot vector clamped vs. open?).
  Out of scope; this is a schema validator, not a solver pre-flight.
- **Issue messages are not byte-equivalent to Python's.** The
  *codes* are byte-equivalent (e.g., both produce
  `LINE_ZERO_LENGTH`), so cross-implementation tests can compare
  on `code` alone. Free-form `message` strings are produced via
  `std::ostringstream` and may format doubles differently than
  Python's f-strings. Documented in `validate()`'s header comment.
